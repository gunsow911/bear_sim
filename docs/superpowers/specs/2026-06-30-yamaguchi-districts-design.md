# 山口市ステージ 地区選定 設計書

作成日: 2026-06-30

## 目的

山口市の e-Stat 小地域（基本単位区）Shapefile を素材に、熊出没シミュレーション
（遭遇率管理ゲーム）の**ステージ「地区」**を選定する。本書は地区の**粒度と分割方針**を
確定するもの。数理パラメータの実データ化（baseDensity 等）は対象外（`docs/DATA.md` 参照）。

## 背景・現状の課題

現行は旧市町村ベースの 7 地区（`src/data/sampleStage.ts` / `scripts/build-geojson.mjs`）。
e-Stat 小地域を S_NAME プレフィックスで束ねている。実測した規模分布は極端に不均一：

| 地区 | 人口 | 面積 | 単位区 |
|---|---:|---:|---:|
| center（旧山口市まるごと） | 132,292 | 304km² | 3,215（445字） |
| ogori | 26,344 | 33km² | 588 |
| ajisu | 9,233 | 25km² | 115 |
| akiho | 8,212 | 40km² | 108 |
| miyano | 7,823 | 36km² | 124 |
| tokuji | 5,196 | 290km² | 143 |
| ato | 4,866 | 293km² | 137 |

**課題**: `center` が全人口の約7割・445字を1ブロックに潰している。本ゲームの主眼である
「里山（供給源）→バッファ→市街縁辺→市街中心」という**遭遇率の勾配**が、その肝心の
領域である `center` の内部に埋もれ、地区単位で見えない。一方、阿東・徳地は人口希薄な
巨大山地で、これは熊の供給源として 1 地区が妥当。

→ **巨大 center を性格の異なる単位に割る**ことが地区選定の核心。

## 採用方針

**10 地区。里山→市街グラデーションで分割。**

旧町 6 つの馴染みある名前は残し、旧山口市 center を性格の違う単位
（中心市街 / 大内 / 吉敷 / 仁保・小鯖里山 / 南部平野）に分割する。
（当初は大内・吉敷を 1 地区＝9 地区案だったが、大内は北東・吉敷は西で中心市街を挟んで
両側に位置し、1 地区にすると必ず飛び地化するため分割して 10 地区とした。
検討した代替案: 現状7地区維持＝勾配が埋もれる / 12地区＝操作対象過多 /
5地区＝地域差が潰れる。いずれも本ゲームの学習目的に対し粒度が不適と判断。）

達し、中心市街へ迫る、という棲み分けの構造を地区の並びと隣接でそのまま表現できる。

## 分割の実装方針（データ契約）

- **入力**: `data_raw/yamaguchi/r2kb35203.shp`（属性 Shift_JIS、4,430 基本単位区）。
- **分割キー**: 各単位区の `S_NAME`（字名）プレフィックスを `districtId` に写像する
  ルールを `scripts/build-geojson.mjs` の `DISTRICT_RULE` に定義し直す。
- **新ルールの骨子**（`S_NAME` をアンカー(^)で判定。上から順に評価、最初に一致を採用）:
  1. `阿東` → `ato`
  2. `徳地` → `tokuji`
  3. `宮野` → `miyano`
  4. `仁保` または `上小鯖`/`下小鯖`/`小鯖` → `niho`
  5. `大内` または `青葉台` → `ouchi`
  6. `吉敷` → `yoshiki`
  7. `小郡` → `ogori`（「小郡上郷仁保津…」もここに含まれる）
  8. `阿知須` → `ajisu`
  9. `名田島`/`佐山`/`嘉川`/`陶`/`鋳銭司`/`秋穂二島`/`秋穂`/`江崎`/`深溝` → `nanbu`
  10. 上記いずれにも一致しない残余 → `center`（＝旧山口市の市街核）
  - **字名バリアントの注意**（実データで判明）: 小鯖は「上小鯖/下小鯖」表記、仁保津は
    「小郡上郷仁保津」表記（=小郡）、青葉台は大内の団地、江崎・深溝は南部沿岸平野。
    これらを取りこぼすと catch-all の center に流れて飛び地化するため明示が必須。
- **ジオメトリ整形（重要）**: e-Stat 基本単位区は隣接境界が完全一致しておらず、
  そのまま dissolve すると無数の隙間（穴）と断片が生じる（実測: 226 穴・1456 断片）。
  対策として `-i ... snap`（取り込み時スナップ）＋ dissolve 後 `-clean gap-fill-area=2km2`
  （残る隙間を最長境界の隣接へ割当）＋生成後 JS で小さな内部穴(<2km²)を埋める。
  結果、各地区はほぼ 1 ポリゴン・穴 0 で市域を隙間なく充填する。
- **dissolve**: `-dissolve2 districtId` で 10 ポリゴンに融合。
- **出力**: `src/data/stages/yamaguchi/districtsGeo.ts`（`FeatureCollection`、各 Feature の
  `properties.districtId` / `name`）。`stage.ts` の地区 id と 1:1 一致させる。
- **隣接トポロジ**: `stage.ts` の `adjacencies` は、生成ジオメトリの**実接触**
  （境界最短距離 < 150m）から作成する。隣接の `features`（water / green-corridor /
  trunk-road）はゲーム意味づけのため地形に基づき付与。連結性（孤立地区が無いこと）は
  テストで担保。

### 残課題（実装時に確定）

- 「中心市街(残余)」に意図しない郊外字が混入していないか、355 大字の全リストを
  目視点検して `DISTRICT_RULE` を微調整する。特に `平川`・`大歳`・`湯田` の市街/郊外の
  線引き、urban 部の `〇〇町` 名（大字でなく町丁名で入る）が center に正しく落ちるか。
- `仁保`・`小鯖` のプレフィックスが実 `S_NAME` と一致するか（合併前町名が
  `仁保下郷` 等の形で入る想定）を、生成後の地区数・人口で検算する。

## 検証（受け入れ条件）

- 生成 GeoJSON の Feature 数がちょうど 10、`districtId` が上表 10 種と一致する。
- 各地区はほぼ 1 ポリゴン・内部の穴 0（市域を隙間なく充填、飛び地が無い）。
- 各地区の合計人口・面積が上表の規模目安と大きく乖離しない
  （`center` が再び 7 割を占める=分割失敗、等を検知）。
- `stage.ts` の `districts[].id` と GeoJSON の `districtId` が 1:1 対応。
- 隣接グラフが連結（孤立地区が無い）であることをテストで担保。
- 地図（`MapView`）に 10 地区が表示され、クリックで個別選択できる。
